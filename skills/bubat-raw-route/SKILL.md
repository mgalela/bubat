# BUBAT Raw Route

Resolve `WORKSPACE_ROOT` first:
- use `.` if `shared/stage-index.md` exists in current directory
- else use `.bubat` if `.bubat/shared/stage-index.md` exists
- else stop and ask user to confirm BUBAT workspace root

1. Read `${WORKSPACE_ROOT}/raw/SOURCES.md` if present.
2. Build candidate source set from:
   - files in `${WORKSPACE_ROOT}/raw/` excluding `README.md`, `MANIFEST.md`, `SOURCES.md`, and `.gitkeep`
   - any file/dir paths registered in `${WORKSPACE_ROOT}/raw/SOURCES.md`
3. If no candidate sources found, offer user option:
   - add files into workspace `raw/`
   - or register external source first with `raw add <path>`
4. For each candidate file, read or inspect its content (parse text-based formats; describe binary/image formats by filename and extension).
5. Determine which stage(s) the file is most relevant to using this routing table:

| Content signals                                                                            | Route to stage        |
| ------------------------------------------------------------------------------------------ | --------------------- |
| Business goals, problem statement, stakeholders, high-level requirements, NFRs, tech stack | `01-discovery`        |
| Business processes, user journeys, workflows, use cases, scenarios                         | `01b-flow`            |
| Domain model, bounded contexts, ubiquitous language, context map                           | `01c-bounded-context` |
| Entity diagrams, ER diagrams, data dictionaries, schema descriptions, attribute lists      | `01d-data-model`      |
| System landscape, external integrations, system-level diagrams                             | `02-context`          |
| Deployment units, services, databases, messaging, infrastructure                           | `03-container`        |
| API specs, component diagrams, internal module structure, code-level design                | `04-component`        |
| Final architecture docs, decision summaries, full system description                       | `05-document`         |
| Cavekit specs, implementation task lists, build plans, OpenAPI/proto/interface skeletons   | `06-spec`             |

6. A file may route to multiple stages if it spans topics — list all relevant stages.
7. Stage matching uses table column `Stages`; each stage may read only rows where `Stages` contains its exact stage id (for example `01-discovery`).
8. Write `${WORKSPACE_ROOT}/raw/MANIFEST.md` using this format:

```markdown
# ${WORKSPACE_ROOT}/raw/MANIFEST.md

<!-- Generated by `raw route`. Edit manually if needed. -->

| File | Stages | Notes |
| ---- | ------ | ----- |
| raw/filename.pdf | 01-discovery, 01b-flow | Contains requirements and process flows |
| docs/spec.yaml | 04-component | OpenAPI spec -- routes and schemas only |
```

9. Present manifest to user for review. Ask if any routing looks wrong before saving.

Notes:
- `File` may point inside `raw/` or registered external source path.
- Prefer relative paths from workspace parent/project root when possible.
- If external source path looks broad or noisy, suggest narrowing path before save.
